## Scope (Phase V-1 only — buildable with zero machines placed)

1. **`/machines` hub page** (public, add to PUBLIC_ROUTES + sitemap + proxy test): the locator shell. Pre-placement state: honest "Machines are coming to [region] — first location announcing soon" + the value prop framed as CONVENIENCE, not deals ("sealed product, in your hands now; restock alerts; scan any machine to price-check") + email capture for "be first to know when a machine lands near you" (new Beehiiv source tag via lib/beehiiv.ts — keep the import boundary). Architect the page so per-location cards slot in at V-2 (a `machines` data module with an empty-but-typed locations array; no DB table needed until real locations exist — decide and document). **Copy rule (amended firewall §4 rule 1): the words "deal," "best price," "below market," "verified" must NOT appear on vending surfaces** — pin this with a structural test like the visual-regression guards.
2. **`/host` venue-funnel page** (public, add to PUBLIC_ROUTES + sitemap + proxy test): the pitch to business owners who'd house a Foil machine. **Build it to the §5b playbook in the strategy doc — it's the researched copy spec, follow it section by section:** lead with zero-cost/zero-work via the asymmetry block ("We do: machine, stock, install, service, insurance, card payments, locator listing, restock announcements. You do: 3 sq ft and a standard outlet"), then amenity + destination-traffic-with-mechanism (locator page per machine, restock alerts pinging nearby collectors — mechanism only, NEVER a traffic or dollar number), then the published deal (**John's verdict, decided 2026-06-12: "10–15% revenue share of gross sales, paid monthly" — use it verbatim, no stop-and-ask needed;** final % within the band is set per venue by traffic), then risk-removal terms (60–90 day trial, 30-day out, service SLA; liability-coverage line only once John confirms the policy exists — do not claim insurance Foil doesn't carry yet, Gate 13). Proof block: founder identity + Level 4 TCGplayer seller credential; NO testimonials until real ones exist (output-skill: no placeholder quotes, ever). Lead-capture form per §5b field list (~6 required: name, business name, venue type dropdown, city, email/phone, daily foot traffic bucketed; optional: hours, placement/outlet, **"do you already sell trading cards?"**, what-matters-most, free text) — store via Server Action (new `host_leads` table, service-role only, RLS on; same shape as watchlist patterns) + notify Discord through lib/notifications/discord.ts (keep the single import boundary; soft-fail). **FTC hard NOs (structural-test-pinned like the deal-vocabulary rule): no host-income projections, no "passive income," no operator-recruitment content on this page.**
3. **Location-anchored restock alerts (schema + server action only, no cron yet):** extend the watchlist/alerts data model so an alert can anchor to a machine location + product (sealed SKU or "any restock") instead of a card slug + target price. Email-anchored like ADR-020 watchlists. Migration in supabase/migrations/ via the standard CLI path. Do NOT build the alert-send path yet — there's nothing to alert on; the schema + signup form on /machines is the deliverable. Keep entitlement-seam compatibility (ADR-059 SUB row).
4. **`/pricing-methodology` — "Machine pricing" disclosure section (amended §4):** the convenience-premium policy in public, stated plainly: machine product is convenience-priced ABOVE market for instant in-person availability; the online deal-finder exists for the best price; machine inventory never influences what the site shows (deal rankings, buy-signal, resolver are structurally blind to it); visible per-customer purchase limits. Gate 13 anti-hype voice rules apply — every claim must be literally true NOW or framed as policy-for-launch, not current fact. This disclosure is the brand's pre-positioned answer to any future "deal site charges $20/pack" criticism.
5. **Homepage: NO changes** (per §5 — "Find a Foil machine" entry waits for machine #1).
